Design Review Decisions

Feedback from screenshot review, Feb 23 2026. Updates to component guides and spec.

Data & Features

1 Assignee Avatars Free

"We didn't talk about this user assignment element. That should be a premium feature, right?"
Decision: Free. Assignees are core task metadata — hiding them behind a paywall makes the free tier feel crippled. Falls under Principle 2: "if required for readability → free."

Avatar colors: Assignee-specific, not phase-colored. Each person gets a consistent color across the entire chart so you can visually scan "where is Sarah's work." Auto-assigned from a palette of 8–10 distinct colors. The colors rotate and stick per user.

Rendering: 22px circle, 2px lighter tint background, 9px bold initials in the assignee color. Appears at the right edge of the sidebar task row.

2 Status Enum — At-Risk / Blocked Changed

"Where did the at risk states come from? Are those in the system?"
Decision: Three default statuses, rest opt-in.

The library ships with 3 built-in statuses: done, active, upcoming. These get automatic visual treatments (colors, opacity, gradients).

at-risk and blocked are available via config but not shown by default. The host app (Leantime) maps its own internal statuses to the library's visual treatments. If Leantime has a "stuck" status, it can map that → blocked visuals.

Updated spec: Remove at-risk and blocked from default examples. Show them only in a "Custom Status Mapping" section.
Free vs Paid Model

3 Vibrant Free, Not Grayscale Confirmed

"Are we doing a grayscale version for the free and then selling colored versions?"
Decision: Free is the full vibrant design. No degradation. No "unlock the colors." Nobody evaluates ugly software and thinks "this will be good once I pay."

Premium templates are alternative complete design systems — different spacing, typography, radius, animations, label treatments. The Presentation theme looks different from Working View. The Dense theme packs more information. The Dark theme inverts everything. None of them are "the same thing but prettier."

The free Working View should be the best Gantt anyone has ever seen out of a free library. That's the adoption engine.
Inline Editing

4 Input Styling — Phase-Colored Border Changed

"Should the inputs have similar rounding? Big plus on the colored outline to match the block. It's not written."
Decision: Yes. When clicking a phase name to edit, the input border color matches the phase color (not always --blu). This creates a visual thread from the color dot to the editing state.

Input specs (updated):
• Border: 1px solid {phaseColor}
• Focus ring: 0 0 0 3px {phaseColor at 12% opacity}
• Border radius: 6px (consistent with system rounding, not as round as pills)
• Font: Same as phase name (13px, weight 600)
• Background: transparent (text floats in place)
• Trigger: single click on name text
• Save: Enter key or blur (click away)
• Cancel: Escape key → restores original
Dependency Arrows — Redesign

5 Arrow Visibility — Darker Dots Changed

"The dots need to be darker, they aren't visible now."
Decision: Connection dots increase from opacity 0.4 to opacity 0.7. Dots grow from 2.5px to 3.5px radius. Color stays --ink4 (#9ca3af) but the higher opacity makes them clearly visible against the light background.

6 Dashed vs Solid — Status-Based Arrow Style New

"Maybe it's a dash when it hasn't started yet, and then solid lines when they are in active work?"
Decision: Yes — arrows reflect dependency status.

Active dependency (source is done or active, target is active):
• Solid line, no dash
• 1px stroke, --ink3 (#6b7280), opacity 0.5
• Connection dot 3.5px, opacity 0.7

Future dependency (target is upcoming):
• Dashed line, stroke-dasharray 4 3
• 1px stroke, --ink4 (#9ca3af), opacity 0.35
• Connection dot 3px, opacity 0.5

Completed dependency (both are done):
• Solid line, very faded
• 1px stroke, --ink5 (#d1d5db), opacity 0.25
• Dot 2.5px, nearly invisible

This creates a natural hierarchy: active connections are clear, future ones are suggested, completed ones fade away.

7 Arrow Routing — Bottom-to-Top Instead of End-to-Start New

"The multi-row curves get really messy, especially if they are totally separated work groups."
Decision: Arrows exit from bottom-center of source bar, enter top-center of target bar.

Why this is better:
• Arrows always flow downward, matching natural reading direction
• No horizontal spaghetti crossing unrelated rows
• Works cleanly even when source and target are far apart vertically
• No collision with bar content (labels sit on bars, arrows go between them)

Path routing:
• Exit: bottom-center of source bar
• Curve: vertical S-curve with control points offset proportional to row distance
• Enter: top-center of target bar
• For adjacent rows: nearly straight vertical drop
• For distant rows: gentle S-curve with wider spread
Alternative considered: Show arrows only on hover (like Linear). Keep in view as a future toggle option, but default to always-visible with the bottom-to-top routing. Users who want clean views can collapse to "no arrows" mode.
Today Marker

8 Z-Index — Above Everything Changed

"The today marker should sit on top of the milestones, not behind it."
Decision: Today marker line and dot render above bars and milestones. It's the most important temporal reference — nothing should occlude it.

Updated z-index stack:
• Column glow: z-index 1 (lowest, behind everything)
• Grid lines: z-index 2
• Bars and milestones: z-index 10
• Dependency arrows: z-index 15
• Today line: z-index 20
• Today dot: z-index 25
• Tooltips: z-index 1000

9 Pulsing Dot — Centered + Ring Changed

"The blinky dot needs to be centered to the line."
Decision: Dot is precisely centered on the line (transform: translate(-50%, -50%) from line's top position). Added an outer ring for more presence.

Updated dot spec:
• Inner dot: 8px, solid --red, no animation
• Outer ring: 16px, --red at 15% opacity, pulses scale(1) → scale(1.6) over 2s
• Ring provides presence without garish blinking of the dot itself
• Both centered on line top position
Drag & Drop

10 Resize Handle Color — Status-Matched Changed

"Does the resize mode match the color of the milestone itself?"
Decision: Yes. When actively dragging a resize handle, the handle border matches the bar's status color (not always blue). The active handle gets: border: 1.5px solid {statusColor}.

Default (hover, not dragging): white handle, neutral shadow.
Active (dragging): white handle + colored border matching bar.

11 Hover Handles — Inset from Edge Changed

"Can the hover handles come in a little bit? They are too on the edge."
Decision: Handles inset 4px from bar edges. This creates a visible gap between the handle and the bar's rounded ends, making handles clearly grabbable without visually merging.

Updated positioning:
• Left handle: left: 4px (was -1px)
• Right handle: right: 4px from track end (was flush)
• Handles sit inside the bar shape, not overlapping edges
